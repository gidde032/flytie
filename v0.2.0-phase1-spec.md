# v0.2.0 Phase 1 — Spec Addendum

Three features in one phase: `undelete`, `stats`, and `material merge`. All are non-schema-changing — no Alembic migration required, no impact on existing databases. A single three-reviewer pass covers all three after implementation.

---

## FR-9 — Undelete

### Requirement

The `flytie undelete <name>` command restores a soft-deleted pattern to active status. This closes the gap documented in handoff.md known-issue #4: once a pattern is soft-deleted, its version history becomes invisible (`versions`/`diff`/`restore` all funnel through `get_pattern(include_deleted=False)`), and there is no way to bring it back without a database export round-trip.

### CLI surface

```
flytie undelete <name>
```

- `<name>` — the pattern's display name (case-insensitive lookup via `name_key`, same as all other commands).
- If the pattern exists and is soft-deleted (`is_deleted=True`): set `is_deleted=False`, update `updated_at`, print `[green]Restored[/green] <name>`.
- If the pattern exists but is **not** deleted: print an informational message — `"<name>" is not deleted` — and exit code 0. Not an error; the user's intent (the pattern should be active) is already satisfied.
- If no pattern matches (including hard-deleted): raise `PatternNotFoundError` → exit code 1 with `"Pattern '<name>' not found"`, same as `view`/`edit`/`delete` for missing patterns.
- No `--yes` / confirmation prompt. Undelete is non-destructive; the pattern simply reappears in listings, search, and `shop`.

### Core logic

Add `undelete_pattern(session, name)` to `core/patterns.py`:
- Call `get_pattern(session, name, include_deleted=True)`.
- If `pattern.is_deleted` is `False`, return `None` (or a sentinel) so the CLI can print the "not deleted" message.
- Otherwise set `pattern.is_deleted = False`, flush, return the pattern.

### Interactions with existing commands

- After undelete, the pattern is visible to `list`, `search`, `view`, `versions`, `diff`, `restore`, `shop`, `export`, `suggest`, `export-db`, and `tag list` — same as any active pattern.
- `delete` on the same pattern soft-deletes it again (idempotent lifecycle: add → delete → undelete → delete → ...).
- `delete --hard` on a soft-deleted pattern still works (permanently removes); `undelete` after a hard delete fails with `PatternNotFoundError`.

### Edge cases

- Name collision: impossible. Soft-delete does not free the `name_key` unique constraint — the row stays in the table with `is_deleted=True`. Undelete just flips the flag back.
- Version history: fully preserved through the soft-delete/undelete cycle. `versions`, `diff`, `restore` all work normally once the pattern is active again.

## FR-10 — Library Statistics

### Requirement

The `flytie stats` command prints a read-only summary of the user's pattern library. No arguments, no flags.

### CLI surface

```
flytie stats
```

Output is a single Rich panel with three sections.

**Library overview:**
- Total active patterns (with soft-deleted count in parentheses if nonzero, e.g. `12 patterns (3 deleted)`)
- Total versions across all active patterns
- Total unique materials
- Total species
- Total tags

**Top 5s** (each as a compact numbered list with counts; show fewer if the library has fewer):
- Most-used materials — by number of active patterns using them
- Most-tagged species — by number of active patterns
- Most-versioned patterns — by version count

**Timeline:**
- Oldest pattern (name + created date)
- Newest pattern (name + created date)
- Most recently edited (name + updated date)
- Average versions per pattern (one decimal)

If the database has no active patterns, print a short message: `No patterns yet — run [bold]flytie add[/bold] to get started.` and exit code 0.

### Core logic

Add `library_stats(session)` to a new module `core/stats.py`, returning a dataclass or NamedTuple with all the computed fields. All queries go through the existing SQLAlchemy session — no new models or schema. Counts exclude soft-deleted patterns except where noted (the deleted count itself, obviously).

"Most-used materials" counts distinct active patterns whose current version references each material via `pattern_materials`, not total `pattern_materials` rows (a material used in 3 versions of the same pattern counts as 1 pattern, not 3).

### Rendering

Add `render_stats(stats)` to `render.py`. Uses `rich.panel.Panel` with `rich.table.Table` subsections to match the visual register of `view` and `shop`.

Dates render as `YYYY-MM-DD` (no time component — the library-summary context doesn't need hour precision).

### Interactions with existing commands

- `stats` calls `_open_db` like every other command, so it respects `validate_compatibility` and requires `flytie init` first.
- Read-only — no writes, no side effects. Safe to run at any time.

## FR-11 — Material Merge

### Requirement

The `flytie material merge` command collapses a source material into a target material, rewriting all `pattern_materials` references. This is the canonical mitigation for the material deduplication problem described in spec §8: `"size 14 dry fly hook"` and `"14 dry hook"` are different strings to the database but the same thing to a tier.

### CLI surface

```
flytie material merge <from> <to> [--dry-run]
```

- `<from>` — the source material's canonical name (case-insensitive).
- `<to>` — the target material's canonical name (case-insensitive). Must already exist in the database.
- `--dry-run` — print what would happen without making changes.
- Both `<from>` and `<to>` must exist. If either is missing, exit code 1 with `"Material '<name>' not found"`.
- If `<from>` and `<to>` resolve to the same material (after normalization), exit code 1 with `"Cannot merge a material into itself"`.

**Output (normal run):**

```
Merged "size 14 dry fly hook" → "dry fly hook" (affected 3 patterns, 5 version rows)
```

The counts tell the user the blast radius. "patterns" = distinct active patterns affected; "version rows" = total `pattern_materials` rows rewritten (including historical versions).

**Output (`--dry-run`):**

```
Would merge "size 14 dry fly hook" → "dry fly hook"
  Affected patterns: Adams, Elk Hair Caddis, Pheasant Tail
  Version rows: 5
```

Lists the affected pattern names so the user can sanity-check before committing.

### Core logic

Add `merge_materials(session, from_name, to_name, dry_run=False)` to `core/patterns.py` (or a new `core/materials.py` if you prefer — implementer's discretion). Returns a result object with the affected pattern names and row count.

Steps:
1. Look up both `Material` rows by canonical name (case-insensitive via `func.lower`).
2. Find all `PatternMaterial` rows where `material_id = from_material.id`.
3. For each, update `material_id` to `to_material.id`. If a version already has a row for `to_material` (the same version references both the source and target material), this is a **duplicate within a version**. Resolution: keep the target row, sum quantities if both have numeric quantities and matching units, discard the source row. If units differ or either quantity is null, keep the target row unchanged and discard the source row (the user can edit manually afterward). Print a warning per affected version: `Warning: <pattern> v<N> had both materials; kept "<to>" row`.
4. Delete the now-orphaned `from_material` row from the `materials` table. (The `ondelete="RESTRICT"` FK on `pattern_materials.material_id` means this only succeeds if step 3 fully cleared the references — a safety net.)
5. If `dry_run`, roll back the transaction (or skip the writes entirely). Otherwise commit.

### Versioning interaction

Material merge rewrites historical version rows. This is intentional — the merge is saying "these were always the same material." It does **not** create new pattern versions. The version history reflects the merge retroactively, which matches the user's mental model: they're correcting a data-entry inconsistency, not making a new edit.

### Edge cases

- **Duplicate within a version** (same version has both `from` and `to` materials): handled by the merge logic above with a per-version warning.
- **Category mismatch**: `from` is category "hook", `to` is category "thread". The merge proceeds — the user is asserting they're the same material, so the target's category wins. No warning needed; the user chose the target.
- **`from` material is referenced by no patterns at all** (orphan): merge still works — just deletes the orphan material row. Output: `Merged "..." → "..." (affected 0 patterns, 0 version rows)`.
- **Export/import round-trip**: after a merge, `export-db` reflects the merged state. An `import-db` of a pre-merge export will re-introduce the old material name as a new `Material` row (expected — the export is a snapshot of the state at export time).

## Testing expectations

All three features should have:
- Happy-path tests for the core logic functions.
- CLI integration tests via `CliRunner`.
- Edge-case coverage for the cases listed above (not-deleted pattern for undelete, empty library for stats, duplicate-within-version for merge, same-material-to-self for merge).
- The review pass will audit coverage; aim for the same density as existing commands.

Test file: `tests/test_v0_2_phase1.py` (or split per feature if the file grows past ~300 lines).

## Out of scope for this phase

- `--alias` as a persistent mapping (e.g., "always treat X as Y on future adds"). The merge is a one-time operation. Persistent aliases are a v0.2+ candidate if the merge pattern proves useful.
- Fuzzy auto-matching of similar material names. The spec §8 mitigation defers this; `merge` is the manual tool.
- `stats --format json` or any non-terminal output format.
- Schema migrations of any kind.
